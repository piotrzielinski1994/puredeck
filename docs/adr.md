# Architectural Decisions - puredeck

Append-only log of architectural and design decisions made during development.

## Format

Each entry follows this structure:

| Date | Decision | Rationale |
|------|----------|-----------|
| {YYYY-MM-DD} | {What was decided} | {Why this choice was made} |

## Decisions

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-07-17 | Desktop shell via Tauri 2 (not Electron) | Smaller bundle, native webview, Rust backend; core platform choice, expensive to swap. Mirrors the `requi` repo's proven stack. |
| 2026-07-17 | Adopt TanStack ecosystem (Router/Query/Table/Form/Hotkeys) | Single coherent stack; permeates the whole frontend architecture. Reuses the pattern validated in `requi`. |
| 2026-07-17 | Keybindings via `@tanstack/react-hotkeys` despite alpha status | Official TanStack lib chosen over stable alternatives; alpha = API-churn risk, isolated behind the command-palette component. |
| 2026-07-17 | Code-based TanStack Router routes (not file-based) | Fewer build plugins for a scaffold; revisit if the route tree grows. |
| 2026-07-17 | App shell drives content by in-app tabs, not routes: single `/` route + `WorkspaceContext` (Settings/Study/decks are tabs, `/settings` route removed) | Mirrors `requi`; enables multiple decks open at once + close/reorder. Reverses the bootstrap's route-per-page direction (supersedes the "revisit as the tree grows" note above) - costly to unwind once features hang off the tab model. |
| 2026-07-17 | Persist layout/theme via a pluggable `SettingsStore` port: `@tauri-apps/plugin-store` in-app, in-memory in dev/test (env-branched) | Deep seam with two real adapters; keeps dev-browser + jsdom deterministic and packaged app durable. Mirrors `requi`. |
| 2026-07-17 | Decks persist as one-file-per-deck on disk (`collections/<slug>.json`) behind a read-only `CollectionStore` port (`load` only, no `save`); demo decks seeded to disk once on first run | Plain hand-editable/back-up-able JSON, no hidden state; read-only mirrors `requi`'s workspace-config deferral (in-app CRUD is a later feature). Costly to unwind once decks live on disk. Uses `@tauri-apps/plugin-fs` because `LazyStore` can't enumerate a directory. |
| 2026-07-17 | Keyboard bindings behind a shortcut-action registry + pure resolver + `useActionHotkeys`; overrides persisted to a separate `keymap.json` store file | Registry is the single source of the action list; overrides are device-syncable on their own and kept out of settings.json. Single-binding (`actionId -> string`) not requi's multi-binding array - only 3 actions, widen later if needed. |
| 2026-07-18 | Reverse the read-only `CollectionStore` (2026-07-17): add `save(deck)` to the port + both adapters; the whole deck file is rewritten atomically on each card edit/add/delete. Tauri adapter tracks a `deckId -> slug` map at load so `save` writes back to the exact file a deck came from (load previously discarded the filename). | In-app card CRUD (F1) is table stakes for a flashcards app; the read-only deferral has been reached. Whole-file rewrite keeps decks plain and hand-editable, no partial-write state; the slug map avoids clobbering a colliding-slug file. Deck-level CRUD (create/rename/delete) still deferred to F3. |
| 2026-07-19 | SRS scheduling (F2) uses classic **SM-2** (not FSRS or Leitner), at **day granularity**. Per-card review state persists to a **separate** `ReviewStore` (`review-state.json`), not embedded in deck JSON. `Again` keeps the card due today and re-queues it within the current session; full Anki sub-minute learning-step timers (1m/10m) are deferred. | SM-2 is a pure, ~40-line, fully unit-testable settled algorithm; FSRS needs 19 tuned weights + a full review-history log, Leitner is too coarse. Separate store keeps deck files content-only and hand-editable, mirroring the keymap.json-separate-from-settings.json seam. Day granularity can't express sub-minute steps; in-session requeue is the faithful single-sitting collapse. Costly to unwind once review state lives on disk in a given shape. |
| 2026-07-19 | **SUPERSEDES the SM-2 row above.** F2 scheduler is **FSRS-6** via the `ts-fsrs` library (runtime dependency, MIT, zero runtime deps), not a hand-rolled SM-2. Persist at **sub-day (ISO datetime) granularity**; interval **fuzz ON** (seeded per card id for deterministic tests); short-term learning-steps path ON. Per-card FSRS state in `review-state.json`; full append-only review history in a new `review-log.json` (`RevlogStore`). Default 21-weight params, retention 0.9, max interval 36500. FSRS optimizer (custom weight training) deferred. | Anki's actual default scheduler since 25.07 is FSRS-6, so "behave exactly like Anki" requires FSRS, not SM-2 (SM-2 is legacy in Anki). `ts-fsrs` is maintained by the same org (open-spaced-repetition) as Anki's Rust engine and tracks the shipped algorithm - far lower risk than hand-porting a 21-weight ML model (anti-hallucination). First runtime dep accepted: the fidelity requirement outweighs the zero-dep ethos, and the package is dependency-free + MIT. Sub-day timestamps + fuzz + short-term path are what make same-day/learning behaviour match Anki; day-granularity (prior row) can't. revlog stored now so a future optimizer/stats (F5) can train on real history - the one algorithm that genuinely consumes history. SM-2 code was never merged, so no data migration. Costly to unwind (scheduler core + on-disk state shape). |
