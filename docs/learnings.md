- `nvm` on this machine is shimmed to `mise`; node is managed by mise (`.nvmrc` pins 24). In non-interactive bash, activate first: `eval "$(mise activate bash)"` then `mise exec -- <cmd>` to get node 24.
- TanStack Hotkeys split: `@tanstack/hotkeys` is the framework-agnostic core (no React hook). The React `useHotkey`/`useHotkeys` + `HotkeysProvider` live in `@tanstack/react-hotkeys`. Install the adapter, not the core.
- Hotkey strings are case-sensitive in the typed union: use uppercase keys, e.g. `"Mod+K"` not `"Mod+k"`.
- Under jsdom the hotkeys lib resolves `Mod` to `Control` (test platform reports non-mac), so hotkey tests fire `{Control>}k{/Control}`, not Meta.
- `npm create tauri-app` can't target a non-empty dir: scaffold in a temp dir, then copy `src-tauri/` + vite/ts configs + `index.html` in, then rewrite identity (Cargo `name`, `tauri.conf.json` productName/identifier, `main.rs` `_lib` ref).
- jsdom has no `ResizeObserver`, no `Element.prototype.scrollIntoView`, and no `window.matchMedia`; radix + cmdk need them. No-op stubs live in `tests/setup.ts`.
- Under jsdom `@tanstack/react-hotkeys` `detectPlatform()` resolves to non-mac, so `Mod` maps to `Control`. Test the `Mod+K` palette by firing `fireEvent.keyDown(document, { key: "k", code: "KeyK", ctrlKey: true })` - the real global-hotkey path (HotkeysProvider -> useHotkey) opens the palette in jsdom, no fallback needed.
- The Vite dev port is 1433 (not the tauri-template default 1420) in BOTH `vite.config.ts` `server.port` and `tauri.conf.json` `build.devUrl` - they must match or the Tauri webview loads a blank page.
- `@tanstack/react-hotkeys` `useHotkey` degrades gracefully WITHOUT a `HotkeysProvider` (`useDefaultHotkeysOptions` is `useContext(...)?.defaultOptions ?? {}`), so components using it (e.g. StudyView `Space`, Main `Mod+B`) are unit-mountable in isolation - no provider wrapper needed in their tests.
- TDD ordering with subagents: spawn the RED test-writer subagent and let it finish (confirm red) BEFORE writing the production code. Running them in parallel means the tests never see red in the working tree - the writer finds the modules already present and has to reconstruct red in a throwaway worktree, which defeats the gate.
- Persistence store is env-branched via `isTauri()` from `@tauri-apps/plugin-store`-adjacent `@tauri-apps/api/core`: `createSettingsStore()` returns the `plugin-store` adapter in the native app and an in-memory one everywhere else (dev-browser + jsdom tests), mirroring requi. Tests that need Tauri absent can `vi.mock("@tauri-apps/api/core", () => ({ invoke: vi.fn(), isTauri: () => false }))`.
- `@tanstack/hotkeys` core `validateHotkey` returns `valid:true` for an unknown key (e.g. `"bogus"` -> only a `"Unknown key: ..."` warning). So `safeNormalize` must treat a warning containing `"Unknown key"` as invalid (return null), not just `valid:false`. Empty string is hard-invalid (`valid:false`, `"Hotkey cannot be empty"`).
- `useActionHotkeys` reads `SettingsContext` directly via `useContext` (context is exported from settings-context.tsx) and falls back to `DEFAULT_SETTINGS.shortcuts` when no provider - keeps StudyView/Main unit-mountable in isolation (same degrade-gracefully property the raw `useHotkey` had). Do NOT switch it to `useSettings()`, which throws without a provider and breaks the bare-mount tests.
- `@tauri-apps/plugin-fs` cannot enumerate a directory from a `LazyStore` (that's the store plugin, single-file). Deck discovery uses `readDir`/`readTextFile`/`writeTextFile`/`mkdir`/`exists` from plugin-fs. Capability needs `fs:allow-read-dir`/`-read-text-file`/`-write-text-file`/`-mkdir`/`-exists` + an `fs:scope` allowing `$APPDATA/**` (both settings.json and collections/ live under app-data). Register `tauri_plugin_fs::init()` in lib.rs.
- Keymap is split into its own `keymap.json` store file (separate `LazyStore`) from `settings.json`, mirroring requi: on save, `shortcuts` is stripped from the settings payload and written to keymap.json; on load both files are read and re-merged. Keeps the settings file focused and the keymap device-syncable on its own.
- SRS scheduling (F2) uses **FSRS-6** via the `ts-fsrs` library (NOT hand-rolled SM-2 - Anki's actual default scheduler since 25.07 is FSRS). A thin seam `src/lib/study/fsrs.ts` wraps the library: `createScheduler()` (default 21-weight params, retention 0.9, max 36500, fuzz on + card-id seed strategy, short-term on), `newCard(now)`, `gradeReview(scheduler, card, rating, cid, now)->{card,log}`. Per-card FSRS state in `review-state.json`, append-only history in `review-log.json` - two SEPARATE LazyStore files, same separate-store-per-concern seam as keymap.json. Deck/Card content model unchanged.
- ts-fsrs persistence: FSRS `Card` has `Date` fields (`due`, `last_review`). JSON.stringify -> ISO strings, but `scheduler.next()` needs real `Date`s, so a bare JSON.parse breaks it. `mergeReviews`/`mergeRevlog` REVIVE ISO->Date on load (and drop entries without a valid due). Route every tauri-adapter load through the merge fn.
- ts-fsrs seeded determinism: `fsrs({enable_fuzz:true})` alone is NOT enough - attach a seed strategy `useStrategy(StrategyMode.SEED, GenSeedStrategyWithCardId('cid'))` and set `card.cid` before `.next()`. To TEST fuzz determinism meaningfully, use a MULTI-DAY interval (scheduled_days>1) - a learning-step due (+10min) is unfuzzed, so asserting it is tautological. `Grade` = `Exclude<Rating, Manual>` (the 4 real ratings), narrower than `Rating`; re-export it, don't alias `Grade=Rating`.
- Time in SRS is INJECTED (`now: Date`), never read via `new Date()` inside pure code. A React component defaulting a prop to `now = nowDate()` creates a NEW Date every render; if an effect deps on it, it loops. Memoize: `const now = useMemo(() => nowProp ?? nowDate(), [nowProp])`. A mount-snapshot queue that must rebuild on late-async data needs an effect + a `hasGraded` ref guard so grading doesn't reset it; and `key={deck.id}` on StudyView so switching study tabs remounts (else session state leaks across decks).
- Same-day/"still due today" comparisons use LOCAL calendar (`getFullYear/getMonth/getDate`), NOT `toISOString().slice(0,10)` (UTC) - a learning-step due crossing UTC midnight would wrongly drop for non-UTC users. Anki uses a local-day cutoff.
- Side effects (store `.save`) must NOT go inside a `setState(updater)` callback - React StrictMode double-invokes updaters and would double-fire the save. Compute next state from current, then `setState(next)` + `save(next)` outside (see `gradeCard`, mirrors `saveDeck`).
